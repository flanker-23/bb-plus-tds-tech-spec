1. Problem Statement

With Account Statements decomp in progress, we are now fetching Account Statements for all merchants (by communicating to the banks) via the new service X-Account-Statements but would also dual writing those statements back to API.

The reads are still all being powered by API monolith.

With Read cutoff, we aim to remove the dependency with API & eventually stop dual write of statements on API.

2. Introduction & Scope 

What is X Account Statements as a product?

Razorpay X powers account statements via X Dashboard where the merchant can view/download/email/filter and search on the account statements. The account statements are fetched on a regular basis for all merchants by razorpay via provider bank.

Merchants also rely on Account Statements report where we maintain an SLA of T+1 to show all transactions of the merchant’s account.

Razorpay also provides out of the box APIs for the merchant to use at their perusal for fetching :

  • Fetching specific transaction
  • Fetching all transactions in a given range[a][b][c][d][e]

Where is Account Statements built and what are we doing with Account Statements?

We are currently decomposing the account statements functionalities built on API to support all the aforementioned features.

6. Current Architecture / Current HLD

6.1 Decomposition Strategy

How are we decomposing Account Statements from API Monolith?

We are using decomposition by Subdomain here to decompose components from API monolith to separate microservices. We have identified domains lying inside API Monolith, and trying to define and create microservices catering to each domain, further creating subdomains inside them while developing (though we’re trying to follow a mix of MVC and DDD to create the microservice)

In this approach, a major domain is decomposed into multiple subdomains and recursively into further smaller subdomains. The major domain here is Banking Related products lying inside API e.g. Account Statements, Payouts, Balances, each of which can be their own microservice.

Major Domain (Banking Monolith)

        |--------- Core Domains e.g. Account Statements, Payouts(Contain business logic)
        |--------- Supporting Domains (Not direct, but still related to business)

        |--------- Generic Domains (Non Business domains e.g. notification service)

What is the strategy for decomposition?

There are two ways we could’ve approached the entire decomposition of Account Statements

  • Develop everything in one go and completely route everything to the new microservice.
  • Rewrite the domain into a new microservice and have API and new microservice coexist till the traffic from monolith has shrunken to the point of elimination.

We are following the second approach, which is a loose implementation of Strangler Fig Pattern for decomposing monoliths.

In stranger fig pattern, we go about the decomposition in 3 steps.

  1. Transform - Create components by rewriting them (e.g. we wrote x-account-statements microservice)
  2. Coexist - Keep the monolith for rollback. Intercept outside system calls by introducing an HTTP proxy (something like an API gateway) & route the traffic to the new system.
  3. Eliminate - Finally, when the new components have existed for a good amount of time without any major unforeseen problems, the monolith can be eliminated completely.

For Account Statements decomposition, we are following a similar structure but without the API gateway proxy layer.

Here’s how the ideal implementation of the strangler fig pattern would look like :

Here, the strategy we’re using for Statements write cutoff is similar to this, except for the API gateway layer.

We’re not routing any request to the new microservice but defined another set of incoming HTTP requests for Ramped up merchants and using the new microservice for their statement processing.

Are we decomposing the entire domain in one go or decomposing in phases of Write Cutoff & Read Cutoff?

We are decomposing in multiple phases :

  • Phase 1: Writes powered by the new Microservice, with still dual writing to the monolith, reads powered by the monolith only for ALL use cases (that’s why dual writing)
  • Phase 2 : Reads to be powered by the new microservice, eventually making the monolith obsolete for the business domain we’re decomposing.

Question[i] : To achieve phase 2, what should be the approach?

  • Do we stop the writes to monolith altogether and let the reads get powered by the microservice?
  • Do we still dual write to the monolith, having a fallback logic in case something goes wrong with getting the data from the new service? (Preferred)
  • What about historic data?
  • In both of these cases, legacy data should be migrated to the new service (more on handling legacy data later in the doc)

6.4 Problems with the current architecture

6.4.1 Reads happening from monolith

Currently, all reads are happening from API database and corresponding datalake instance.

This is a problem since we want to completely eliminate monolith from the picture. [l]


We want to decouple all use cases of reads happening from these sources, completely cutting off the monolith.

6.4.2 Multiple domains are queried along with statements

There are multiple sources where we’ve exposed endpoints that provide the capability to query on account statements with filters from other business domain e.g. payouts, contacts, fund accounts etc. [m]

This poses a problem as it poses challenges when coming to decomposition and introduces tackling latency issues.

6.4.3 Near realtime data vs SLA

For Account statements, the SLA that is agreed with merchants is T+1, majorly for account statements reports where the merchant can see all their transactions at T+1.

However, we have endpoints and X-Dashboard that attempts to show near real time data, which is not guaranteed.

Banks do not give real time data of the transactions anyway and we attempt to fetch the account statements on an interval basis. Since this is dependent on the banks, adding complexity to the system by trying to provide real time transactions is not [n][o]suggested.[p]

7. Final Approach - Specifications[aa]

7.1 Cutting off monolith sources

To completely cut off monolith sources, we want to tackle the following sections and provide solutions for each of them.

7.1.1 Account Statements Data

Currently all account statements data is dual written to monolith sources. To provide all data of account statements, we have two sub categories :

  • New statements being fetched after Write Cutoff : New Microservice can be the source
  • Older statements before write cutoff : Migration

7.1.1.1 Data Migration[ab]

We would need to migrate the data present in API monolith to X-Account-Statements database to be able to read the statements for dates before the merchant was onboarded to X-Account-Statements

Since we’re planning to start read cutoff after 100% merchant rollout to XAS, data set of statements to be migrated would be static. All new statements would already be there on XAS DB.

Assuming cutoff time for each merchant to be 1st March of the previous year (to support FY transactions), date range and query to get all statements to migrate would be

select * from banking_account_statements where merchant_id = ‘MID’ AND created_date > ‘FYEndingDate’

We can download the results and update the XAS statements table via 2 ways

  • Have a way to upload csv to insert all rows into the table
  • Have an API exposed at XAS side, which can read the csv and insert into batches (preferred)

Possibility of Cold Storage?

We can also introduce a cold storage like glacier to read the data for older statements.

7.1.2.1.2 Requirement of Read Replica in XAS

Since we would be read heavy on XAS post the read cutoff, we want all read queries to be routed to a read replica instance of the database.

Few things are required for this :

  • Read replica setup for XAS DB [not present (slack)]
  • Support in XAS codebase to route queries to read on demand. [Development task]
  • While creating new APIs on XAS, we need to make sure the DB queries happening are routed to read replicas of the instance.

7.1.2.2 Central Query Service[at]

Another solution to the problem is to have a Centralized Query service which can access data from multiple domains (payouts, account statements, contacts) and act as a layer to cater to all read use cases. [au][av]

Later the same service can be used to support read cutoff for other business domains where querying across subdomains is required.

Microservice for Account Statements would sync all writes of the microservice table to the centralised database, and the same would go for tables that are still written by API monolith

CQS (Centralised Query Service) would intercept all the incoming requests and directly query on the centralised database to get the data.

This solution will also implement CQRS pattern where the writes would be owned by the microservice (Command : INSERT/UPDATE), and the reads would be powered by the Central Query Service (Query : GET)

Overall the system would look something like this :

7.1.2.2.1 Pros

  1. Single source of truth for all queries[aw]
  2. Can serve as a query system for other business domains with read cutoffs where cross domain queries are necessary
  3. Reduced latency

7.1.2.2.2 Cons

  1. Single point of failure
  2. Additional service to maintain[ax]
  3. Routing of all endpoints to the Central query service (Includes Edge onboarding)[ay]

7.1.2.2.3 How real time do we want the data to be?

All account statements data does not have to be real time[az], since bank does not provide us data of the transactions in real time, we do not need to guarantee real time data.
Only SLA is T+1 reports that merchants need to download.

However, this does not apply to syncing of data from XAS DB to synced centralised storage of the CQS as there is one use case where payouts reads account statements data before changing the status of the payout.
If the statement is present, it should be returned with minimal lag.

7.1.2.2.3.1 Can the same condition apply to other domains to be decomposed out of monolith?

Not necessarily, for other services being decomposed out of monolith, there might be requirements of near real-time data (eg listing payouts)

We need to factor this if we want to extend the use case of CQS to other business domains which are to be decomposed in future.

Other solution : Keep CQS only for querying purpose where not real-time data is required (<1s of latency)
Can think of CQS as a service that can serve near real-time data.

7.1.2.2.4.1.3 Latency Requirements?

How fresh should the data be in the CQS?

  • Real-time (<1s)?
  • Near real-time (a few seconds to minutes)?
  • Eventual consistency (minutes to hours)?


Based on our previous findings and assumptions, we want near real time data for the reads.
While real-time data should be served directly from the decomposed microservices via API calls for sensitive use cases.

7.1.2.2.4.1.4 Data Volume and Growth?

What’s the size of data expected?

  • How fast will it grow (GBs/day or TBs/month)?
  • Currently, our microservice is growing at a rate of 1.6GB/day
  • Any retention policies?
  • We can think of moving <1 year old data to cold storage, we do not need to retain all the data in CQS since account statements are not required frequently for data older thant the financial year.
  • Source events (if stored in CQS) can be evicted after 15 days (Do not see any use case of storing the source events though)

7.1.2.2.4.1.5 Consistency Guarantees?

Do we need strong consistency (ACID) for the reads?

Or is eventual consistency acceptable (especially across domains)?

We do not require strong ACID for the reads

  • Account statements are historical records that are typically immutable once created
  • The data represents past transactions that have already been processed and settled
  • Updates to existing statements are rare and usually only happen during reconciliation (enrichment) or error correction

7.1.2.2.4.1.6 Mutability of the records?[ba]

For account statements, records are going to change only once (in case of enrichment) and if the record is external, then never. [bb]
We do not need to care for heavily mutable data here.

Note: Not necessarily the case for other business subdomains which can leverage CQS.

7.1.2.2.4.1.7 Data Source Ownership
  • Will CQS be the owner of data stored?
  • No, the data in CQS is only meant for read purpose without strong consistency.
  • Individual microservices would be the owner of the data they’ll push to CQS for effective querying purposes.

7.1.2.2.4.1.8 Cost Constraints?

We do not have a cost constraint, but would be evaluating the possible databases on this aspect as well.

7.1.2.2.4.1.9 Monitoring and Observability?

We want monitoring capability on the database. We would need slow query tracings as well, and to fix them once we identify.

7.1.2.2.7 Final solution

Based on the aforementioned parameters, ElasticSearch seems to be the best possible solution for Warm Storage for the Central Query Service as it performs well against all factors that we want for Account Statements use case.

Reasons :

  • Our use case is for eventual consistency
  • VERY LOW latency & excellent search performance
  • We want to support fuzzy searching (contact name, email, etc)
  • Our searchable fields are not mutable
  • If we flatten the data and push to ES, we would not require reindexing in case the document gets updated
  • Flexible schema can account for new fields being added to the searchable index.
  • Flatten Document removes the requirement for JOINS
  • Non mutable data removes the con for cost of indexing

7.1.2.2.8 Can we remove CQS and only use the database as a central warm storage across all microservices for their read use cases?

There are two possible architectures with either a Central Query service layer to query the warm and cold storages, or multiple microservices querying the central storages for data.

Central Query Service Architecture

Shared Central Database Architecture

7.1.2.2.8.1 Pros

Advantage

Explanation

Simpler Architecture

No need to build and maintain a separate CQS service layer with its own APIs, auth, scaling, etc.

Lower Latency

Eliminates the extra network hop and faster responses in some cases (direct DB access).

Lesser Development

Can start consuming data from day one without needing to define/query via an API layer.

Uniform Interface

All services can use SQL directly to query the DB — no need to reimplement logic in CQS APIs.

Lower Infra Cost

No extra service running = fewer machines and memory, assuming DB can handle the load.

Fewer Code Paths

Debugging/monitoring one shared DB rather than also tracking behavior through CQS logic.

7.1.2.2.8.1 Cons

Concern

Explanation

Tight Coupling

Microservices tightly depend on DB schema, breaking DB changes becomes risky (though will rarely happen)

Access control

Currently we do not have any such auth control over which database can be accessed by which microservice, but still a concern with distributed microservice architecture. Eg. Account Statements Service should not have access to read PII data from any table that is pushing sensitive information.

Access control is a concern.

Duplication of Logic

Each microservice might write its own query logic (joins, filters, aggregates), leading to inconsistencies.

CQS can enable endpoints which can be reused by microservices for their use cases.

You also miss out on Domain level APIs, e.g. we would miss introducing high-level read intents like getEnrichedStatementForMerchant as cleanly as an API would return.

Performance

With CQS in picture, we can cache frequent queries at CQS layer.
If all microservices start querying on the shared database, performance can take a hit.

Query structure lies with each microservice

Without a central service, you lose control over query shapes, inefficiencies, or slow reads.

Query Optimization

CQS could batch, cache, or rewrite inefficient queries. Without it, performance tuning is harder.

And all query tunings would need to be taken care of at individual microservice layer.

Error Handling Becomes Tricky

In CQS, we can centralize timeouts, fallbacks, error patterns. Without it, error behavior varies per microservice.

How is the data fetched in the API when these filters are passed?


The system uses a combination of joins and relationships to fetch this data.

Base Query Structure :

SELECT * FROM banking_account_statement

USE INDEX (banking_account_statement_merchant_id_created_at_index)

WHERE entity_id IS NOT NULL

The repository adds joins based on the required fields :

E.g. for contacts related information

JOIN fund_accounts ON fund_accounts.id = payouts.fund_account_id

JOIN contacts ON contacts.id = fund_accounts.source_id AND fund_accounts.source_type = 'contact'

Lazy loading relationships

The repository defines several expand relationships that are loaded when needed:

protected $expandsForTypePayout = [

    'source.fundAccount.contact',  // Loads contact details

    'source.fundAccount.account',  // Loads account details

    'source.reversal',            // Loads reversal details

];

protected $expandsForTypePayoutForDashboard = [

    'source.fundAccount.contact',

    'source.fundAccount.account',

    'source.balance'

];

Query Building Process

The repository has specific methods for each field:

  protected function addQueryParamContactId(BuilderEx $query, array $params)

  protected function addQueryParamContactName(BuilderEx $query, array $params)

  protected function addQueryParamContactEmail(BuilderEx $query, array $params)

  protected function addQueryParamContactPhone(BuilderEx $query, array $params)

  protected function addQueryParamPayoutId(BuilderEx $query, array $params)

Final Query Structure

When all the joins are added, the final query looks something like this:

SELECT banking_account_statement.*

FROM banking_account_statement

USE INDEX (banking_account_statement_merchant_id_created_at_index)

LEFT JOIN payouts ON payouts.id = banking_account_statement.entity_id

    AND banking_account_statement.entity_type = 'payout'

LEFT JOIN fund_accounts ON fund_accounts.id = payouts.fund_account_id

LEFT JOIN contacts ON contacts.id = fund_accounts.source_id

    AND fund_accounts.source_type = 'contact'

WHERE banking_account_statement.entity_id IS NOT NULL

Should we support all such filters while trying to decompose the Account Statements?

Point to note here is that since this is monolithic application, it is able to access multiple tables like Payouts, Fund Accounts, Contacts etc. to get data from if the specific filter is passed on to the API.

But if we consider the microservice, it would not have direct access to query the other tables, and only have it’s own x-account-statements.account_statements table to query the statement data from.
