BB+ TDS Integration: API Versioning Recommendation

Date: February 2026

Author: Architecture Review

Status: Draft for Discussion

1. Executive Summary

Recommendation: Extend Existing /v2/payouts (No New Version)

Based on analysis of the current architecture, public API documentation, and UI flows, we recommend extending the existing internal /v2/payouts API with optional TDS fields rather than creating a new API version.

  • No v3 API needed - TDS fields are already internal-only
  • Backward compatible - All new fields are optional
  • Minimal Payouts changes - Pre-compute pattern keeps logic in Tax-Compliance
  • Works for all channels - UI, Bulk, and future API exposure

3. API Versioning Options

3.1 Option A: Create New /v3/payouts API

Not Recommended 
POST /v3/payouts
{
  "fund_account_id": "fa_xxx",
  "gross_amount": 1000000,                  // NEW: Gross amount
  "currency": "INR",
  "mode": "IMPS",
  "purpose": "vendor bill",

  "tds": {
    "category": "194C",                     // Category code
    "source": "bb_plus",                    // Source indicator
    "is_ldc_applicable": true               // LDC flag
  }
  // net_amount computed by Payouts internally
}

Problems with v3 Approach

  • Breaking change: amount → gross_amount semantic change
  • Payouts computes TDS: Requires S2S call to Tax-Compliance in critical path
  • Domain leakage: Payouts must understand TDS computation
  • Maintenance burden: Three API versions to maintain (v1, v2, v3)
  • Migration cost: Dashboard/BFF must migrate to v3
  • Latency: Adds 50-200ms to payout creation

3.2 Option B: Extend /v2/payouts with Optional TDS Fields

Recommended 
POST /v2/payouts
{
  "fund_account_id": "fa_xxx",
  "amount": 900000,                         // NET amount (caller computes)
  "currency": "INR",
  "mode": "IMPS",
  "purpose": "vendor bill",

  "tds": {
    // Existing fields (Vendor-Payments)
    "category_id": 14,
    "amount": 100000,

    // NEW: BB+ fields (all optional)
    "source": "bb_plus",                    // Routing indicator
    "gross_amount": 1000000,                // For audit trail
    "breakdown": {                          // From compute response
      "basic_tax_amount": 90000,
      "surcharge": 0,
      "cess": 10000,
      "effective_rate": 10.0,
      "category_code": "194C",
      "is_ldc_applied": false
    }
  }
}

Benefits of Extension Approach

  • Backward compatible: All new fields optional
  • No semantic change: amount remains NET
  • No S2S call: Caller pre-computes TDS
  • Domain separation: Tax-Compliance owns computation
  • Minimal changes: Just add optional fields + topic routing
  • Single API: No new version to maintain

3.3 Option C: Separate /v1/tds/payouts API (Tax-Compliance Owns)

Alternative 
// Tax-Compliance API
POST /v1/tds/payouts
{
  "fund_account_id": "fa_xxx",
  "gross_amount": 1000000,
  "category": "194C",
  "is_ldc_applicable": true,
  "mode": "IMPS",
  "purpose": "vendor bill"
}

// Tax-Compliance:
// 1. Computes TDS
// 2. Calls Payouts /v2/payouts internally
// 3. Returns combined response

Considerations

  • Pro: Clean separation - Tax-Compliance is single entry point
  • Pro: TDS Payment created synchronously
  • Con: Tax-Compliance becomes payout orchestrator (domain leakage)
  • Con: Complex for bulk payouts
  • Con: Tight coupling - Tax-Compliance depends on Payouts API

4. Detailed Recommendation

4.1 Extend /v2/payouts (No New Version)

Aspect Decision Rationale
API Version Keep /v2/payouts TDS is internal; no public API change needed
Amount Semantics amount = NET Consistent with existing behavior; caller sends net
TDS Computation Pre-compute (caller calls Tax-Compliance first) No S2S call in payout creation; domain separation
New TDS Fields All optional Backward compatible; Vendor-Payments unchanged
Kafka Routing Based on tds.source BB+ → Tax-Compliance; Others → Vendor-Payments
Public API (/v1) Unchanged External merchants don't use TDS (yet)

4.2 TDSDetails Field Extension

// internal/app/dtos/payoutDetails.go

type TDSDetails struct {
    // ═══════════════════════════════════════════════════════════
    // EXISTING FIELDS (Vendor-Payments - unchanged)
    // ═══════════════════════════════════════════════════════════
    TDSCategoryID nulls.UInt32  `json:"category_id" binding:"omitempty"`
    TDSAmount     nulls.Float64 `json:"amount" binding:"omitempty"`

    // ═══════════════════════════════════════════════════════════
    // NEW FIELDS (BB+ - all optional, pre-computed by caller)
    // ═══════════════════════════════════════════════════════════

    // Routing indicator: "bb_plus" routes to Tax-Compliance topic
    Source string `json:"source" binding:"omitempty"`

    // Gross amount for audit trail (amount field is NET)
    GrossAmount nulls.Int64 `json:"gross_amount" binding:"omitempty"`

    // Full breakdown from Tax-Compliance compute response
    Breakdown *TDSBreakdown `json:"breakdown" binding:"omitempty"`
}

type TDSBreakdown struct {
    BasicTaxAmount float64 `json:"basic_tax_amount"`
    Surcharge      float64 `json:"surcharge"`
    Cess           float64 `json:"cess"`
    EffectiveRate  float64 `json:"effective_rate"`
    CategoryCode   string  `json:"category_code"`    // "194C"
    IsLdcApplied   bool    `json:"is_ldc_applied"`
    LdcRate        float64 `json:"ldc_rate,omitempty"`
    MajorHead      int     `json:"major_head"`
    MinorHead      int     `json:"minor_head"`
}

// IsBBPlusFlow returns true if this is a BB+ TDS payout
func (t *TDSDetails) IsBBPlusFlow() bool {
    return t.Source == "bb_plus"
}

4.3 Kafka Topic Routing

// internal/app/payouts/core.go

const (
    TdsTopic       = "add-tds-entry"          // Existing → Vendor-Payments
    TdsTopicBBPlus = "add-tds-entry-bb-plus"  // NEW → Tax-Compliance
)

func (c Core) ProcessTdsForPayout(ctx context.Context, payout *Payout, oldStatus string) errors.IError {
    if MoneyTransferredStatus[payout.GetStatus()] {
        payoutDetails := payout.GetPayoutDetails(ctx)
        additionalInfo := payoutDetails.GetAdditionalInfo()

        // ════════════════════════════════════════════════════
        // ONLY CHANGE: Route based on source
        // ════════════════════════════════════════════════════
        topic := TdsTopic
        if additionalInfo.GetTDSSource() == "bb_plus" {
            topic = TdsTopicBBPlus
        }

        message := c.buildTdsMessage(ctx, payout, payoutDetails, additionalInfo)
        eventClient := provider.GetEventClient(ctx)
        eventClient.PublishEvent(message, topic)
    }
    return nil
}

8. Conclusion

Final Recommendation: Extend /v2/payouts

  1. No new API version - Extend existing /v2/payouts with optional TDS fields
  2. Pre-compute pattern - BFF/Dashboard calls Tax-Compliance compute first
  3. Payouts stores only - No computation logic in Payouts
  4. Topic routing - tds.source = "bb_plus" → Tax-Compliance Kafka topic
  5. Public API unchanged - /v1/payouts remains as-is (for now)

This approach provides:

  • Minimal risk to high-throughput Payouts API
  • Clean domain separation
  • Works for UI, Bulk, and future API exposure
  • Backward compatible with Vendor-Payments flow
