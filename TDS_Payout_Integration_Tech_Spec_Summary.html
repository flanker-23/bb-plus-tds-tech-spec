1. Overview & Problem Statement

1.1 Business Context: Why TDS Support?

BB+ merchants need to comply with Indian tax regulations when making vendor payments. When TDS (Tax Deducted at Source) applies:

  • Merchant enters a gross amount (e.g., ₹10,000)
  • Selects a TDS category (e.g., Section 194J - Professional Services at 10%)
  • Optionally applies Lower Deduction Certificate (LDC) for reduced rates
  • System calculates TDS amount (₹1,000) and creates a payout for the net amount (₹9,000) to the vendor
  • TDS amount is tracked for monthly filing and payment to the government

1.2 Product Requirement

Seamless Merchant Experience

The product requirement is to provide a seamless experience for merchants using the Payouts API, where they can pass minimal details similar to what is currently passed to the compute API.

Key Requirements:

  • Minimal Input: Merchants should provide basic payout details (gross amount, TDS category, contact, etc.) without needing to pre-calculate TDS values themselves
  • Automatic Calculation: The system should automatically compute TDS amounts based on the provided inputs
  • Persistence: All TDS-related data should be persisted correctly for compliance and reporting
  • Unified Flow: The payout should be created seamlessly after TDS calculation
  • Universal Applicability: This single approach should work consistently across all three payout types:
    • Single Payout (UI-based merchant-initiated payouts)
    • Bulk Payout (CSV file upload with multiple payout records)
    • API Payout (Programmatic payout creation via API integration)

This requirement drives the architectural decision to ensure merchants have a consistent, simple interface regardless of how they create payouts, while the system handles all complexity of TDS computation and compliance internally.

1.3 The Problem

Current State

The existing Payouts service does not support BB+ TDS computation and tracking. We need to integrate TDS functionality while:

  • Not impacting existing Vendor-Payments flows (which have their own TDS handling)
  • Supporting all payout types: Single Payout (UI), Bulk Payout (CSV), and API Payouts
  • Maintaining clear separation of concerns between Payouts and Tax-Compliance services
  • Ensuring data consistency between payout records and TDS filing records

1.4 Services Involved

Service Responsibility
Tax-Compliance
  • Computes TDS based on category, rate, and LDC status
  • Manages TDS Payment entities for compliance and filing
  • Validates TDS categories and PAN details
Payouts
  • Creates payout records for net amount to vendor
  • Processes fund transfers
  • Emits payout status updates
  • Stores TDS information for audit trail
Dashboard/BFF
  • Orchestrates UI flows
  • Calls APIs and presents data to merchants

1.4 Amount Relationship

Key Formula

Gross Amount = Net Amount + TDS Amount

Example: Section 194J (10% rate)
  Gross Amount (vendor payment):  ₹10,000
  TDS Amount (10%):               ₹1,000
  Net Amount (payout to vendor):  ₹9,000

2. Approach 1: Inline Compute Pattern

Not Recommended Payout API accepts TDS inputs; Payouts calls Tax-Compliance internally (S2S); async TDS entity creation

2.1 How It Works

In this approach, the Payouts service accepts TDS computation inputs (gross amount, category, contact details) and internally calls the Tax-Compliance service to compute TDS before creating the payout.

Flow Summary

  1. Preview (UI only): Dashboard calls Tax-Compliance compute API to show TDS breakdown to merchant
  2. Create Payout: Dashboard sends TDS inputs to Payouts API with gross amount
  3. Inline Compute: Payouts makes S2S call to Tax-Compliance compute API
  4. Amount Derivation: Payouts derives net amount (gross - TDS) and stores payout with net amount
  5. Async TDS Entity: When payout reaches terminal state, Payouts publishes to Kafka; Tax-Compliance creates TDS Payment entity

2.3 Payout Type Considerations

Payout Type Impact
Single Payout (UI) One S2S call per payout (+50-200ms latency)
Bulk Payout (CSV) S2S call for every row during payout creation (high latency for bulk operations)
API Payouts External API users pay latency cost; service coupling creates reliability risk

2.4 Pros and Cons

ProsCons
Single API call for merchant/caller S2S call adds 50-200ms latency to every payout
Payout and TDS data stored atomically Payouts fails if Tax-Compliance is down (tight coupling)
Simple integration for frontend Domain leakage - Payouts must understand TDS computation
Amount mutation (gross→net) is risky and error-prone
TDS Payment entity created asynchronously (not in API response)
High testing complexity (must mock Tax-Compliance in all tests)

Why Not Recommended

This approach introduces significant technical debt: latency overhead, service coupling, domain leakage, and risky amount mutation. The synchronous S2S call in the critical path makes payouts dependent on Tax-Compliance availability.

3. Approach 2: Tax-Compliance Orchestrated

Alternative Tax-Compliance orchestrates both TDS computation and payout creation

3.1 How It Works

In this approach, Tax-Compliance becomes the orchestrator. It exposes a single API endpoint that creates both the TDS Payment entity and the payout.

Flow Summary

  1. Preview (UI only): Dashboard calls Tax-Compliance compute API to show TDS breakdown
  2. Single API Call: Dashboard calls Tax-Compliance payout creation API (not Payouts directly)
  3. Tax-Compliance Orchestration:
    • Computes TDS
    • Creates TDS Payment entity (synchronously)
    • Calls Payouts API (S2S) to create payout
    • Returns both TDS Payment ID and Payout ID in response
  4. Status Sync: Tax-Compliance listens to payout status updates via existing Kafka topic

When to Consider

This approach is viable if having the TDS Payment ID synchronously in the API response is a hard requirement. However, it introduces orchestration complexity and reverses the natural dependency direction (Tax-Compliance calling Payouts).

4. Approach 3: Pre-Compute Pattern (RECOMMENDED)

RECOMMENDED - All Channels BFF/Dashboard pre-computes TDS; Payouts receives pre-computed values; minimal changes

Core Principle: Keep Payouts Dumb About TDS Computation

Payouts should NOT call Tax-Compliance during payout creation. Instead, the caller (Dashboard/BFF/API client) pre-computes TDS values by calling Tax-Compliance first, then passes the pre-computed values to Payouts. Payouts simply stores these values and routes to the correct Kafka topic.

4.1 How It Works

This approach separates TDS computation from payout creation, following the same pattern already used by Vendor-Payments.

Flow Summary

  1. Compute TDS (Before Payout): Caller (Dashboard/BFF/API) calls Tax-Compliance compute API
  2. Display Preview: Show merchant the TDS breakdown (gross, TDS, net)
  3. Create Payout: Caller sends pre-computed values to Payouts API
    • amount: NET amount (e.g., 900000)
    • tds.amount: TDS amount (e.g., 100000)
    • tds.gross_amount: Gross amount (e.g., 1000000)
    • tds.source: "bb_plus" (routing indicator)
    • tds.breakdown: Full TDS calculation details
  4. Payouts Storage: Payouts stores payout (net amount) + all TDS fields in payout_details
  5. Kafka Routing: When payout reaches terminal state, Payouts checks source field:
    • If "bb_plus" → publish to add-tds-entry-bb-plus (Tax-Compliance consumer)
    • Otherwise → publish to add-tds-entry (Vendor-Payments consumer - existing)
  6. Async TDS Entity: Tax-Compliance consumer creates TDS Payment entity from Kafka message

6. Final Recommendation

RECOMMENDED: Approach 3 — Pre-Compute Pattern

Approach 3 is the recommended solution for all payout types (Single, Bulk, API).

Key Reasons:

  • Zero latency overhead: No S2S call in payout creation critical path
  • Failure isolation: Payouts doesn't fail if Tax-Compliance is down
  • Minimal Payouts changes: Only add fields + topic routing (low risk, low effort)
  • Clean domain separation: Tax-Compliance owns computation; Payouts owns payout processing
  • No amount mutation: Caller sends NET amount directly (less error-prone)
  • Easy testing: No need to mock Tax-Compliance in Payouts tests
  • Proven pattern: Same approach used by existing Vendor-Payments flow
  • Backward compatible: All new fields are optional; no API version change needed
  • Scales to all channels: Works uniformly for UI, Bulk CSV, and API integrations

6.1 Trade-offs Accepted

Trade-off Mitigation
Two API calls required (compute, then create) BFF/Dashboard handles orchestration; merchants see single flow in UI
TDS Payment ID not available in payout API response Tax-Compliance creates entity asynchronously; ID available via query API if needed
LDC certificate could expire between compute and payout Unlikely in practice (seconds between calls); can add staleness check if needed

6.2 Implementation Priority

  1. Phase 1: Payouts service changes (fields + topic routing) - Low risk, high value
  2. Phase 2: Tax-Compliance compute API + Kafka consumer - Independent of Payouts
  3. Phase 3: Dashboard UI integration (single payout) - First user-facing feature
  4. Phase 4: Bulk payout CSV integration - Reuses all prior work
  5. Phase 5: API documentation for external integrations - Completes all channels

7. Non-Functional Requirements (Summary)

7.1 Performance

MetricTarget
Tax-Compliance compute API latency < 200ms (p95)
Payout creation latency (no change) < 500ms (p95)
Kafka publish to TDS Payment creation < 5 seconds

7.2 Throughput

  • Single Payouts: Support existing payout throughput (no degradation)
  • Bulk Payouts: Support up to 10,000 rows per CSV file
  • Compute API: Handle peak load during bulk processing

7.3 Availability

  • Tax-Compliance compute API: 99.9% uptime
  • Kafka consumer: Automatic retry with DLQ for failed messages
  • Graceful degradation: If compute API down, preview fails but old payouts (without TDS) continue to work

7.4 Data Retention

  • Payout details (TDS fields): Retained per existing payout retention policy (7+ years for compliance)
  • TDS Payment entities: Retained indefinitely for tax filing and audit
  • Kafka messages: 7 days retention minimum

8. Security & Observability (Summary)

8.1 Security

AspectImplementation
PII Handling
  • Deductee PAN encrypted at rest in Tax-Compliance
  • No PAN in Payouts service (only contact_id reference)
  • No PII in logs or Kafka messages
Access Control
  • Tax-Compliance compute API: Authenticated merchants only
  • Payouts API: Existing authentication (no change)
  • TDS Payment query API: Merchant can only access their own records
Kafka Security
  • Use existing Kafka authentication and authorization
  • Topic ACLs to restrict publisher/consumer access

8.2 Observability

ComponentMetrics & Monitoring
Tax-Compliance Compute API
  • Request count, latency (p50, p95, p99)
  • Error rate by error type
  • LDC application rate
Payouts API
  • TDS payout count (source = bb_plus)
  • Kafka publish success/failure rate
Kafka Consumer
  • Consumer lag
  • Message processing rate
  • DLQ message count
  • TDS Payment creation success/failure rate
Alerting
  • Compute API error rate > 5%
  • Kafka consumer lag > 1000 messages
  • DLQ messages present
  • Payout-to-TDS-Payment reconciliation gaps

8.3 Error Handling

  • Compute API errors: Return clear error messages (e.g., invalid category, PAN not found, LDC expired)
  • Payout creation errors: Same as existing payout errors (no change)
  • Kafka publish failures: Use outbox pattern for at-least-once delivery
  • Kafka consumer failures: Retry with exponential backoff; send to DLQ after max retries
  • Reconciliation: Daily job to detect payouts with TDS but no TDS Payment entity (alert ops team)
