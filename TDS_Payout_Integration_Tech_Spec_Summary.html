1. Problem Statement

  • BB+ merchants must comply with Indian tax regulations (TDS) when making vendor payments via Payouts API.
  • The existing Payouts service has no TDS compute or tracking capability for BB+ flows.
  • Vendor-Payments already uses Payouts TDS fields — any change must not break that existing flow.
  • TDS deduction amount must be known at payout creation time for correct money routing; it cannot be computed post-creation.
  • The solution must work uniformly across all three payout channels: Dashboard (UI), Bulk CSV, and Direct API.

2. Introduction & Scope

This document specifies the technical approach for integrating TDS (Tax Deducted at Source) support into the Payouts service for BB+ merchants. It covers the problem, three candidate approaches, the recommended solution, and all downstream considerations.

2.1 Tenets

  • Payouts stays dumb about TDS computation — no S2S call to Tax-Compliance inside the payout creation path.
  • Tax-Compliance owns TDS logic — computation, validation, entity lifecycle, and filing.
  • Vendor-Payments flows are never impacted — all BB+ TDS events route to a separate Kafka topic.
  • Backward compatible by default — all new Payouts fields are optional.
  • No PII in logs or Kafka messages — PAN stored and encrypted only in Tax-Compliance.

2.3 Scope

In scope for this initiative:

  • TDS compute, preview, and payout creation for BB+ merchants — Single Payout (UI), Bulk CSV, and API
  • Payouts service: add TDS fields + Kafka topic routing (source = bb_plus)
  • Tax-Compliance: compute API, Kafka consumer, tds_payments entity creation
  • Tax-Compliance as BFF: preview flow, orchestration for UI/Bulk channels

1.2 Product Requirement

Seamless Merchant Experience

The product requirement is to provide a seamless experience for merchants using the Payouts API, where they can pass minimal details similar to what is currently passed to the compute API.

Key Requirements:

  • Minimal Input: Merchants should provide basic payout details (gross amount, TDS category, contact, etc.) without needing to pre-calculate TDS values themselves
  • Automatic Calculation: The system should automatically compute TDS amounts based on the provided inputs
  • Persistence: All TDS-related data should be persisted correctly for compliance and reporting
  • Unified Flow: The payout should be created seamlessly after TDS calculation
  • Universal Applicability: This single approach should work consistently across all three payout types:
    • Single Payout (UI-based merchant-initiated payouts)
    • Bulk Payout (CSV file upload with multiple payout records)
    • API Payout (Programmatic payout creation via API integration)

This requirement drives the architectural decision to ensure merchants have a consistent, simple interface regardless of how they create payouts, while the system handles all complexity of TDS computation and compliance internally.

1.1 Current Problem

Current State

The existing Payouts service does not support BB+ TDS computation and tracking. We need to integrate TDS functionality while:

  • Not impacting existing Vendor-Payments flows (which have their own TDS handling)
  • Supporting all payout types: Single Payout (UI), Bulk Payout (CSV), and API Payouts
  • Maintaining clear separation of concerns between Payouts and Tax-Compliance services
  • Ensuring data consistency between payout records and TDS filing records

3. Out of Scope

  • TDS filing, government remittance, or challan generation (owned by Tax-Compliance)
  • Changes to Vendor-Payments TDS flows (must remain untouched)
  • TDS support for non-BB+ merchants in this phase
  • Real-time TDS Payment ID in the payout API response (entity is created asynchronously)
  • LDC staleness enforcement between compute and payout creation (accepted trade-off)

4. Futuristic Scope

  • Expose TDS Payment ID in payout webhook/response (requires sync entity creation)
  • LDC staleness check between compute and payout creation
  • Bulk TDS filing dashboard for BB+ merchants
  • Extend to non-BB+ merchants using the same Pre-Compute Pattern
  • TDS compute caching per merchant/category for performance

5. Assumptions, Goals & Non-Goals

5.1 Assumptions

  • Tax-Compliance service will expose or enhance a compute API endpoint accessible by BB+ merchants (external) and by Tax-Compliance BFF (internal)
  • Payouts service will not integrate Tax-Compliance — all compute happens outside the payout creation path
  • The Kafka topic add-tds-entry-bb-plus is new and owned by Tax-Compliance consumer
  • The source = "bb_plus" field in the TDS payload is the routing signal — no other routing logic needed in Payouts
  • LDC certificate validity between compute and payout creation is acceptable (seconds gap in practice)

5.2 Goals

  • Enable TDS deduction and tracking for BB+ payouts across all three channels
  • Zero latency added to the Payouts API critical path
  • Minimal, isolated changes to Payouts (fields + routing only)
  • Clean domain ownership: Tax-Compliance owns compute and tds_payments entity
  • Backward compatible — existing Payouts and Vendor-Payments flows unchanged

5.3 Non-Goals

  • Building TDS compute logic inside the Payouts service
  • Synchronous creation of tds_payments entity at payout creation time
  • Changing any Vendor-Payments Kafka topics or consumer logic
  • Modifying the public /v1/payouts API (changes go to internal /v2/payouts only)

7. Current Architecture / Current HLD

Current State — No BB+ TDS Support

The Payouts service currently has a tds struct in TDSDetails used exclusively by Vendor-Payments. It contains category_id and amount. When a payout with source = vendor_payments reaches a terminal state, Payouts publishes to add-tds-entry Kafka topic, consumed by Vendor-Payments to create a TDS record.

There is no equivalent flow for BB+. Tax-Compliance has no Kafka consumer for payouts, no TDS compute endpoint used at payout time, and no tds_payments entity creation triggered by payout events.

ComponentCurrent StateWhat's Missing for BB+
Payouts TDS fields category_id + amount (Vendor-Payments only) source, gross_amount, breakdown fields for BB+
Kafka routing All TDS payouts → add-tds-entry BB+ payouts → add-tds-entry-bb-plus
Tax-Compliance Compute API exists for preview; no Kafka consumer for payouts Kafka consumer for add-tds-entry-bb-plus + entity creation
BFF / UI flow No BB+ TDS payout creation flow Tax-Compliance to orchestrate compute + payout creation for UI/Bulk

8. Final Approach — Specifications

Selected: Approach 3 — Pre-Compute Pattern

Tax-Compliance (BFF) pre-computes TDS before calling Payouts. Payouts receives pre-computed values, stores them as primitive fields, and routes to the correct Kafka topic. Tax-Compliance creates the tds_payments entity asynchronously via Kafka consumer. Payouts never calls Tax-Compliance.

8.1 List of Possible Solutions

Three approaches were evaluated. Only the final selected approach (Approach 3) is detailed here. Approaches 1 and 2 are documented below for reference.

ApproachSummaryS2S in Payout?Verdict
1. Inline Compute Payouts calls Tax-Compliance synchronously during payout creation Yes — Payouts→Tax-Compliance Not Recommended
2. TC Orchestrated Tax-Compliance calls Payouts (reversed S2S); creates TDS entity synchronously Yes — Tax-Compliance→Payouts Alternative only
3. Pre-Compute Caller pre-computes TDS via Tax-Compliance, passes values to Payouts; no S2S No ✓ Selected

Approach 1: Inline Compute Pattern (Not Recommended)

Not Recommended Payout API accepts TDS inputs; Payouts calls Tax-Compliance internally (S2S); async TDS entity creation

How It Works

The Payouts service accepts TDS computation inputs (gross amount, category, contact details) and internally calls the Tax-Compliance service to compute TDS before creating the payout.

Flow Summary

  1. Preview (UI only): Tax-Compliance (BFF) computes TDS to show breakdown to merchant
  2. Create Payout: Tax-Compliance sends TDS inputs to Payouts API with gross amount
  3. Inline Compute: Payouts makes S2S call to Tax-Compliance compute API
  4. Amount Derivation: Payouts derives net amount (gross - TDS) and stores payout with net amount
  5. Async TDS Entity: When payout reaches terminal state, Payouts publishes to Kafka; Tax-Compliance creates TDS Payment entity

Bulk Payout (CSV) S2S call for every row during payout creation (high latency for bulk operations)
API Payouts External API users pay latency cost; service coupling creates reliability risk

Pros and Cons

Why Not Recommended

This approach introduces significant technical debt: latency overhead, service coupling, domain leakage, and risky amount mutation. The synchronous S2S call in the critical path makes payouts dependent on Tax-Compliance availability.

Approach 2: Tax-Compliance Orchestrated (Alternative)

Alternative Tax-Compliance orchestrates both TDS computation and payout creation

How It Works

Tax-Compliance becomes the orchestrator. It exposes a single API endpoint that creates both the TDS Payment entity and the payout.

Flow Summary

  1. Preview (UI only): Tax-Compliance (BFF) computes TDS to show breakdown
  2. Single API Call: UI calls Tax-Compliance payout creation API (not Payouts directly)
  3. Tax-Compliance Orchestration:
    • Computes TDS
    • Creates TDS Payment entity (synchronously)
    • Calls Payouts API (S2S) to create payout
    • Returns both TDS Payment ID and Payout ID in response
  4. Status Sync: Tax-Compliance listens to payout status updates via existing Kafka topic

This approach is viable if having the TDS Payment ID synchronously in the API response is a hard requirement. However, it introduces orchestration complexity and reverses the natural dependency direction (Tax-Compliance calling Payouts).

Approach 3: Pre-Compute Pattern — SELECTED

RECOMMENDED - All Channels Tax-Compliance (BFF) pre-computes TDS; Payouts receives pre-computed values; minimal changes

BFF = Tax-Compliance Only

In BB+ TDS flows, Tax-Compliance is the only BFF. There is no separate Dashboard/BFF service. The UI talks to Tax-Compliance; Tax-Compliance orchestrates compute and payout creation.

Core Principle: Keep Payouts Dumb About TDS Computation

Payouts should NOT call Tax-Compliance during payout creation. Instead, the caller (Tax-Compliance for UI flows, or API client for programmatic) pre-computes TDS by calling Tax-Compliance first, then passes the pre-computed values to Payouts. Payouts simply stores these values and routes to the correct Kafka topic.

How It Works

This approach separates TDS computation from payout creation, following the same pattern already used by Vendor-Payments.

Flow Summary

  1. Compute TDS (Before Payout): Tax-Compliance (BFF) computes TDS internally; for API clients, they call Tax-Compliance compute API
  2. Display Preview: Tax-Compliance shows merchant the TDS breakdown (gross, TDS, net)
  3. Create Payout: Tax-Compliance (or API client) sends pre-computed values to Payouts API
    • amount: NET amount (e.g., 900000)
    • tds.amount: TDS amount (e.g., 100000)
    • tds.gross_amount: Gross amount (e.g., 1000000)
    • tds.source: "bb_plus" (routing indicator)
    • tds.breakdown: Full TDS calculation details
  4. Payouts Storage: Payouts stores payout (net amount) + all TDS fields in payout_details
  5. Kafka Routing: When payout reaches terminal state, Payouts checks source field:
    • If "bb_plus" → publish to add-tds-entry-bb-plus (Tax-Compliance consumer)
    • Otherwise → publish to add-tds-entry (Vendor-Payments consumer - existing)
  6. Async TDS Entity: Tax-Compliance consumer creates TDS Payment entity from Kafka message

Flow Diagram

Channel-by-Channel Flow

Core Design Tension — API Payouts

In Dashboard and Bulk flows, TDS compute values are known before the payout request is made — Tax-Compliance (BFF) computes and passes them in. For direct API Payouts, the merchant only has raw inputs (category, PAN, amount). TDS compute values are not yet available.

If Payouts calls Tax-Compliance synchronously for API payouts: adds latency (+~200ms p95), creates availability coupling (Payouts fails if Tax-Compliance is down), and increases Payouts blast radius (new S2S integration). This is rejected.

Resolution: Tax-Compliance exposes a public compute endpoint. API merchants make two calls (compute → create payout), passing pre-computed values — identical to the Dashboard/Bulk pattern. Payouts never calls Tax-Compliance.

Channel TDS values available at payout time? Who computes? Payouts change
Dashboard (UI) YES — user filled form, compute shown as preview Tax-Compliance (BFF) Accept + store 4 fields; route to Kafka topic
Bulk CSV YES — computed row-by-row during parse, preview before confirm Tax-Compliance (BFF)
API (direct merchant) NO — merchant has only raw inputs (category, PAN, amount) Merchant calls Tax-Compliance compute API first

Storage Decision

  • Payouts stores: Primitive fields only — tds_amount, tds_rate, tds_category, payee_pan, source (new columns on payout row). No new entity, no Tax-Compliance integration.
  • Tax-Compliance stores: Full tds_payments entity, created asynchronously from the Kafka event after payout completion. Kafka event carries the primitive fields.

The net payout amount is determined by the pre-computed tds_amount at creation time, so money routing is correct before the async entity creation.

Final Decision: Approach 3 — Pre-Compute Pattern

Approach 3 is selected for all channels (Dashboard, Bulk CSV, API). It adds zero latency to payout creation, fully decouples Payouts from Tax-Compliance, requires only low-risk additive changes to Payouts, and works uniformly across all payout types. Trade-offs (two API calls, async TDS Payment ID) are accepted and mitigated.

8.6 Business Logic Changes

8.6.1 Payouts Service — Kafka Routing Logic

The only business logic change in Payouts is inside ProcessTdsForPayout() in core.go. When a payout reaches a terminal state, the function checks the tds.source field to decide which Kafka topic to publish to:

Routing Decision in ProcessTdsForPayout()

  • If tds.source == "bb_plus" → publish to add-tds-entry-bb-plus (new topic, consumed by Tax-Compliance)
  • Otherwise → publish to add-tds-entry (existing topic, consumed by Vendor-Payments — unchanged)
  • If tds.source is empty or not "bb_plus" → existing behaviour, no change
Change AreaWhat ChangesImpact
TDSDetails struct Add optional fields: source (string), gross_amount (int64), breakdown (JSON) Additive — no existing callers affected
Payout API handler Accept new optional TDS fields from request body; store in payout_details JSON No version change to /v2/payouts
ProcessTdsForPayout() Add if tds.source == "bb_plus" branch → publish to new Kafka topic with enriched payload Else-branch untouched; Vendor-Payments routing unchanged
Kafka message BB+ enrichment fields (bb_plus_tds_input, bb_plus_tds_computed) added with omitempty — nil for all non-BB+ payouts Backward-compatible; existing consumers ignore new fields

What Payouts Does NOT Do

  • Payouts does not call Tax-Compliance — no S2S call added
  • Payouts does not validate the pre-computed tds_amount — caller's responsibility
  • Payouts does not mutate the payout amount (caller sends NET amount directly)
  • Payouts does not create any tds_payments entity — Tax-Compliance does that asynchronously

8.6.2 Final Recommendation Summary

RECOMMENDED: Approach 3 — Pre-Compute Pattern

Approach 3 is the recommended solution for all payout types (Single, Bulk, API).

Key Reasons:

  • Zero latency overhead: No S2S call in payout creation critical path
  • Failure isolation: Payouts doesn't fail if Tax-Compliance is down
  • Minimal Payouts changes: Only add fields + topic routing (low risk, low effort)
  • Clean domain separation: Tax-Compliance owns computation; Payouts owns payout processing
  • No amount mutation: Caller sends NET amount directly (less error-prone)
  • Easy testing: No need to mock Tax-Compliance in Payouts tests
  • Proven pattern: Same approach used by existing Vendor-Payments flow
  • Backward compatible: All new fields are optional; no API version change needed
  • Scales to all channels: Works uniformly for UI, Bulk CSV, and API integrations

8.6.3 Trade-offs Accepted

Trade-off Mitigation
Two API calls required (compute, then create) Tax-Compliance (BFF) handles orchestration; merchants see single flow in UI
TDS Payment ID not available in payout API response Tax-Compliance creates entity asynchronously; ID available via query API if needed
LDC certificate could expire between compute and payout Unlikely in practice (seconds between calls); can add staleness check if needed

8.7 Miscellaneous Questions

  • What if Tax-Compliance compute API is down? — Preview fails; no new BB+ TDS payouts can be created until it recovers. Existing non-TDS payouts are unaffected.
  • What if Kafka publish fails? — Outbox pattern ensures at-least-once delivery. tds_payments entity is created on retry.
  • How do we detect payout-to-tds_payment gaps? — Daily reconciliation job in Tax-Compliance detects BB+ payouts (via Payouts query API) that have no corresponding tds_payments entity.
  • Does Payouts validate the pre-computed tds_amount? — No. Payouts trusts the caller (Tax-Compliance BFF or API merchant). Validation is the caller's responsibility.

13.1 Production Rollout and Ramp Plan

  2. Phase 2: Tax-Compliance compute API + Kafka consumer - Independent of Payouts
  3. Phase 3: Tax-Compliance UI flow (single payout) - First user-facing feature
  4. Phase 4: Bulk payout CSV integration - Reuses all prior work
  5. Phase 5: API documentation for external integrations - Completes all channels

9. Non-Functional Requirements (NFRs)

9.1 Scalability

  • Single Payouts: no degradation to existing throughput — Payouts change is additive (new fields only)
  • Bulk Payouts: support up to 10,000 rows per CSV file; Tax-Compliance computes TDS per row before any payout is created
  • Tax-Compliance compute API: must handle peak load during bulk processing concurrently

9.2 Availability

  • Tax-Compliance compute API: 99.9% uptime target
  • Graceful degradation: if compute API is down, preview fails; Payouts creation of non-TDS payouts is unaffected
  • Kafka consumer: automatic retry with exponential backoff; DLQ after max retries
  • Payout creation latency target: < 500ms p95 (no change from current)
  • TDS Payment creation (async): Kafka publish → entity creation < 5 seconds

9.3 Security

AspectImplementation
PII Handling
  • Deductee PAN encrypted at rest in Tax-Compliance
  • No PAN in Payouts service (only contact_id reference)
  • No PII in logs or Kafka messages
Access Control
  • Tax-Compliance compute API: Authenticated merchants only
  • Payouts API: Existing authentication (no change)
  • TDS Payment query API: Merchant can only access their own records
Kafka Security
  • Use existing Kafka authentication and authorization
  • Topic ACLs to restrict publisher/consumer access

9.4 Compliance

  • TDS Payment entities retained indefinitely for tax filing and audit
  • Payout details (TDS fields) retained per existing Payouts retention policy (7+ years)
  • Kafka messages: minimum 7-day retention
  • PAN data: encrypted at rest in Tax-Compliance; never stored in Payouts or Kafka messages

9.5 Reliability

  • Outbox pattern for Kafka publish — at-least-once delivery guaranteed
  • Daily reconciliation job: detect BB+ payouts with no corresponding tds_payments entity and alert ops
  • Payouts → Tax-Compliance are fully decoupled — failure in Tax-Compliance does not affect payout creation

9.6 Infra Cost

  • New Kafka topic: minimal cost (messages are lightweight)
  • Additional Tax-Compliance Kafka consumer: standard pod sizing
  • No new database tables in Payouts — storage via existing payout_details JSON column
