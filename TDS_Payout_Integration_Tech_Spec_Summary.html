1. Problem Statement

  • BB+ merchants must comply with Indian tax regulations (TDS) when making vendor payments via Payouts API.
  • The existing Payouts service has no TDS compute or tracking capability for BB+ flows.
  • Vendor-Payments already uses Payouts TDS fields — any change must not break that existing flow.
  • TDS deduction amount must be known at payout creation time for correct money routing; it cannot be computed post-creation.
  • The solution must work uniformly across all three payout channels: Dashboard (UI), Bulk CSV, and Direct API.

2. Introduction & Scope

This document specifies the technical approach for integrating TDS (Tax Deducted at Source) support into the Payouts service for BB+ merchants. It covers the problem, three candidate approaches, the recommended solution, and all downstream considerations.

2.1 Tenets

  • Payouts stays dumb about TDS computation — no S2S call to Tax-Compliance inside the payout creation path.
  • Tax-Compliance owns TDS logic — computation, validation, entity lifecycle, and filing.
  • Vendor-Payments flows are never impacted — all BB+ TDS events route to a separate Kafka topic.
  • Backward compatible by default — all new Payouts fields are optional.
  • No PII in logs or Kafka messages — PAN stored and encrypted only in Tax-Compliance.

2.3 Scope

In scope for this initiative:

  • TDS compute, preview, and payout creation for BB+ merchants — Single Payout (UI), Bulk CSV, and API
  • Payouts service: enrichment hook for TDS computation
  • Tax-Compliance: internal enrichment API, event listener, tds_payments entity management
  • Dashboard UI and Bulk CSV: TDS form fields and preview

2.4 Product Requirement

Seamless Merchant Experience

The product requirement is to provide a seamless experience for merchants using the Payouts API, where they can pass minimal details similar to what is currently passed to the compute API.

Key Requirements:

  • Minimal Input: Merchants should provide basic payout details (gross amount, TDS category, contact, etc.) without needing to pre-calculate TDS values themselves
  • Automatic Calculation: The system should automatically compute TDS amounts based on the provided inputs
  • Persistence: All TDS-related data should be persisted correctly for compliance and reporting
  • Unified Flow: The payout should be created seamlessly after TDS calculation
  • Universal Applicability: This single approach should work consistently across all three payout types:
    • Single Payout (UI-based merchant-initiated payouts)
    • Bulk Payout (CSV file upload with multiple payout records)
    • API Payout (Programmatic payout creation via API integration)

This requirement drives the architectural decision to ensure merchants have a consistent, simple interface regardless of how they create payouts, while the system handles all complexity of TDS computation and compliance internally.

3. Out of Scope

  • TDS filing, government remittance, or challan generation (owned by Tax-Compliance)
  • Changes to Vendor-Payments TDS flows (must remain untouched)
  • TDS support for non-BB+ merchants in this phase
  • Real-time TDS Payment ID in the payout API response (entity is created asynchronously)
  • LDC staleness enforcement between compute and payout creation (accepted trade-off)

4. Futuristic Scope

  • TDS Payment ID is already included in payout response (enrichment pattern creates entity synchronously)
  • Bulk TDS filing dashboard for BB+ merchants
  • Extend to non-BB+ merchants using the same Enrichment Hook Pattern
  • TDS compute caching per merchant/category for enrichment performance optimization
  • Batch enrichment API for bulk payout optimization (enrich multiple rows in single call)

5. Assumptions, Goals & Non-Goals

5.1 Assumptions

  • Tax-Compliance service will expose or enhance a compute API endpoint accessible by BB+ merchants (external) and by Tax-Compliance BFF (internal)
  • Payouts service will not integrate Tax-Compliance — all compute happens outside the payout creation path
  • The Kafka topic add-tds-entry-bb-plus is new and owned by Tax-Compliance consumer
  • The source = "bb_plus" field in the TDS payload is the routing signal — no other routing logic needed in Payouts
  • LDC certificate validity between compute and payout creation is acceptable (seconds gap in practice)

5.2 Goals

  • Enable TDS deduction and tracking for BB+ payouts across all three channels
  • Zero latency added to the Payouts API critical path
  • Minimal, isolated changes to Payouts (fields + routing only)
  • Clean domain ownership: Tax-Compliance owns compute and tds_payments entity
  • Backward compatible — existing Payouts and Vendor-Payments flows unchanged

5.3 Non-Goals

  • Building TDS compute logic inside the Payouts service
  • Synchronous creation of tds_payments entity at payout creation time
  • Changing any Vendor-Payments Kafka topics or consumer logic
  • Modifying the public /v1/payouts API (changes go to internal /v2/payouts only)

7. Current Architecture / Current HLD

Current State — No BB+ TDS Support

The Payouts service currently has a tds struct in TDSDetails used exclusively by Vendor-Payments. It contains category_id and amount. When a payout with source = vendor_payments reaches a terminal state, Payouts publishes to add-tds-entry Kafka topic, consumed by Vendor-Payments to create a TDS record.

There is no equivalent flow for BB+. Tax-Compliance has no Kafka consumer for payouts, no TDS compute endpoint used at payout time, and no tds_payments entity creation triggered by payout events.

ComponentCurrent StateWhat's Missing for BB+
Payouts TDS fields category_id + amount (Vendor-Payments only) source, gross_amount, breakdown fields for BB+
Kafka routing All TDS payouts → add-tds-entry BB+ payouts → add-tds-entry-bb-plus
Tax-Compliance Compute API exists for preview; no Kafka consumer for payouts Kafka consumer for add-tds-entry-bb-plus + entity creation
BFF / UI flow No BB+ TDS payout creation flow Tax-Compliance to orchestrate compute + payout creation for UI/Bulk

8. Final Approach — Specifications

Selected: Enrichment Hook Pattern (Minimal Inline Compute)

Payouts API accepts optional TDS fields (category_id, gross_amount, apply_ldc). If present, Payouts makes a lightweight synchronous call to Tax-Compliance enrichment endpoint (~50-100ms). Tax-Compliance computes TDS, creates tds_payments entity, and returns enriched data (net_amount, tds_amount, rate, tds_payment_id). Payouts uses net_amount for payout creation. Single API call for merchants; minimal changes to Payouts.

Approach 1: Inline Compute Pattern (Not Recommended)

Not Recommended Payout API accepts TDS inputs; Payouts calls Tax-Compliance internally (S2S); async TDS entity creation

How It Works

The Payouts service accepts TDS computation inputs (gross amount, category, contact details) and internally calls the Tax-Compliance service to compute TDS before creating the payout.

Flow Summary

  1. Preview (UI only): Tax-Compliance (BFF) computes TDS to show breakdown to merchant
  2. Create Payout: Tax-Compliance sends TDS inputs to Payouts API with gross amount

Approach 1: Enrichment Hook Pattern — SELECTED

RECOMMENDED - All Channels Payouts accepts optional TDS fields; makes lightweight sync enrichment call if present; single API for merchants

Core Principle: Single API Call with Minimal Payouts Changes

Payouts API accepts optional TDS fields (category_id, gross_amount, apply_ldc). If these fields are present, Payouts makes a lightweight synchronous call (~50-100ms) to Tax-Compliance enrichment endpoint. Tax-Compliance computes TDS, creates tds_payments entity, and returns enriched data. Payouts uses the enriched net_amount for payout creation. Merchants make ONE API call; zero domain leak to Payouts.

How It Works

This approach adds a minimal, optional enrichment hook in Payouts. When TDS fields are absent, Payouts behaves exactly as before (zero impact). When TDS fields are present, Payouts delegates computation to Tax-Compliance and uses the enriched response.

Flow Summary

  1. API Request: Merchant calls Payouts API with optional TDS fields:
    • amount: Gross amount (e.g., 1000000)
    • tds_category_id: Category ID (e.g., "194J")
    • apply_ldc: true/false (Lower Deduction Certificate)
    • contact_id: For PAN extraction
  2. Enrichment Hook (if TDS fields present): Payouts calls Tax-Compliance internal enrichment API:
    • Input: {contact_id, gross_amount, category_id, apply_ldc}
    • Tax-Compliance: Extracts PAN, computes TDS, creates tds_payments entity (status: PENDING)
    • Output: {net_amount, tds_amount, rate, ldc_applied, tds_payment_id}
  3. Payout Creation: Payouts creates payout with:
    • amount: net_amount (from enrichment)
    • metadata: {tds_payment_id, tds_amount, rate, category_id}
  4. Event Emission: Payouts emits status change events (existing mechanism)
  5. TDS Entity Update: Tax-Compliance listens to payout events and updates tds_payments entity (PENDING → SUCCESS/FAILED)

Channel-by-Channel Flow

Solution to API Payouts Challenge

For API Payouts, merchants only have raw inputs (category, PAN, amount). They don't have pre-computed TDS values. The enrichment hook pattern solves this elegantly:

Merchants make ONE API call to Payouts with TDS fields (gross_amount, category_id, apply_ldc). Payouts internally calls Tax-Compliance enrichment endpoint (~50-100ms), gets the enriched data, and creates the payout. The latency impact (~50-100ms) is acceptable for the following reasons:

  • Isolated Impact: Only affects TDS-enabled payouts; non-TDS payouts have zero latency change
  • Lightweight Operation: Enrichment is fast (compute + DB insert)
  • Scalable: Tax-Compliance can be horizontally scaled
  • One-Stop Solution: Merchants get what they asked for - single API call
Channel How It Works Who computes? Payouts change
Dashboard (UI) UI calls Payouts with TDS fields → Enrichment hook triggers → Creates payout Tax-Compliance (via enrichment endpoint) Enrichment hook (~20 LOC); calls Tax-Compliance if TDS fields present
Bulk CSV For each row: Call Payouts with TDS fields → Enrichment hook → Creates payout Tax-Compliance (via enrichment endpoint)
API (direct merchant) Merchant calls Payouts with TDS fields → Enrichment hook → Creates payout Tax-Compliance (via enrichment endpoint)

Storage Decision

  • Payouts stores: Payout with net_amount + metadata fields — tds_payment_id, tds_amount, tds_rate, category_id (stored in payout metadata/details). No new entity in Payouts.
  • Tax-Compliance stores: Full tds_payments entity, created synchronously during enrichment with status PENDING. Later updated to SUCCESS/FAILED via payout status events.

The net payout amount is determined by the enrichment response at creation time, ensuring correct money routing immediately.

Final Decision: Approach 1 — Enrichment Hook Pattern

Approach 1 is selected for all channels (Dashboard, Bulk CSV, API). It provides a one-stop solution where merchants make a single API call to Payouts with optional TDS fields. Payouts makes a lightweight enrichment call (~50-100ms) to Tax-Compliance if TDS fields are present. This adds minimal code to Payouts (~20 LOC), maintains clean domain separation (Payouts doesn't own TDS logic), and works uniformly across all payout types. The TDS Payment entity is created synchronously and available immediately in the response.

Trade-offs accepted: +50-100ms latency for TDS payouts only (zero impact on non-TDS payouts); sync dependency on Tax-Compliance during payout creation (mitigated via timeouts, circuit breakers, and horizontal scaling).

See Section 8 for full implementation specification.

8.2 Assumptions

  • Tax-Compliance will expose an internal /internal/tds/enrich endpoint for Payouts to call
  • Enrichment call latency is acceptable (~50-100ms) for TDS payouts
  • Tax-Compliance can be horizontally scaled to handle enrichment load
  • Payouts API version stays at /v1/payouts — all TDS fields (tds_category_id, apply_ldc) are optional extensions
  • Enrichment hook is feature-flagged and can be rolled out gradually
  • Tax-Compliance event listener is idempotent — safe to retry on failure
  • Timeouts and circuit breakers will be implemented to handle Tax-Compliance failures gracefully

8.6 Business Logic Changes

8.6.1 Payouts Service — Kafka Routing Logic

The only business logic change in Payouts is inside ProcessTdsForPayout() in core.go. When a payout reaches a terminal state, the function checks the tds.source field to decide which Kafka topic to publish to:

Routing Decision in ProcessTdsForPayout()

  • If tds.source == "bb_plus" → publish to add-tds-entry-bb-plus (new topic, consumed by Tax-Compliance)
  • Otherwise → publish to add-tds-entry (existing topic, consumed by Vendor-Payments — unchanged)
  • If tds.source is empty or not "bb_plus" → existing behaviour, no change
Change AreaWhat ChangesImpact
TDSDetails struct Add optional fields: source (string), gross_amount (int64), breakdown (JSON) Additive — no existing callers affected
Payout API handler Accept new optional TDS fields from request body; store in payout_details JSON No version change to /v2/payouts
ProcessTdsForPayout() Add if tds.source == "bb_plus" branch → publish to new Kafka topic with enriched payload Else-branch untouched; Vendor-Payments routing unchanged
Kafka message BB+ enrichment fields (bb_plus_tds_input, bb_plus_tds_computed) added with omitempty — nil for all non-BB+ payouts Backward-compatible; existing consumers ignore new fields

What Payouts Does NOT Do

  • Payouts does not call Tax-Compliance — no S2S call added
  • Payouts does not validate the pre-computed tds_amount — caller's responsibility
  • Payouts does not mutate the payout amount (caller sends NET amount directly)
  • Payouts does not create any tds_payments entity — Tax-Compliance does that asynchronously

8.6.2 Final Recommendation Summary

RECOMMENDED: Approach 1 — Enrichment Hook Pattern

Approach 1 is the recommended solution for all payout types (Single, Bulk, API).

Key Reasons:

  • One-stop solution: Merchants make single API call to Payouts; enrichment happens internally
  • Minimal Payouts changes: ~20 LOC enrichment hook; feature-flagged; isolated code path
  • Clean domain separation: Payouts doesn't own TDS logic; just calls enrichment service
  • Works for all payout types: Dashboard, Bulk, API - unified approach
  • TDS entity available immediately: Created synchronously during enrichment; returned in response
  • Clean failure handling: Fail fast if enrichment fails; no partial state
  • Low risk: Feature-flagged; zero impact on non-TDS payouts; can be rolled out gradually
  • Scalable: Tax-Compliance enrichment is lightweight and can be horizontally scaled
  • Backward compatible: All TDS fields are optional; no API version change needed

8.6.3 Trade-offs Accepted

Trade-off Mitigation
+50-100ms latency for TDS payouts Isolated to TDS payouts only; zero impact on non-TDS payouts; Tax-Compliance enrichment is lightweight (compute + DB insert)
Sync dependency on Tax-Compliance during payout creation Timeouts (~2s), circuit breakers, retry logic; Tax-Compliance horizontally scalable; fail-fast if unavailable
Enrichment call adds S2S integration to Payouts Isolated behind feature flag; can be disabled; only ~20 LOC; does not affect core payout logic

8.7 Miscellaneous Questions

  • What if Tax-Compliance enrichment API is down? — Payout creation fails immediately with clear error message. No partial state created. Non-TDS payouts continue to work normally.
  • What if enrichment call times out? — Payout creation fails after timeout (~2s). Circuit breaker opens after repeated failures to prevent cascading issues. Merchants retry when Tax-Compliance is back.
  • What if payout creation fails after enrichment? — tds_payments entity remains in PENDING state. Tax-Compliance cleanup job can mark orphaned PENDING records as FAILED after timeout period.
  • What if payout status event is lost? — Eventual consistency: Tax-Compliance can poll Payouts API for payout status and update tds_payments entity accordingly. Daily reconciliation job detects and fixes gaps.
  • How do we monitor enrichment health? — Track enrichment call latency, failure rate, and timeout rate. Alert if p95 latency > 150ms or failure rate > 1%.
  • Does Payouts validate the pre-computed tds_amount? — No. Payouts trusts the caller (Tax-Compliance BFF or API merchant). Validation is the caller's responsibility.

9. Non-Functional Requirements (NFRs)

9.1 Scalability

  • Single Payouts: no degradation to existing throughput — Payouts change is additive (new fields only)
  • Bulk Payouts: support up to 10,000 rows per CSV file; Tax-Compliance computes TDS per row before any payout is created
  • Tax-Compliance compute API: must handle peak load during bulk processing concurrently

9.2 Availability

  • Tax-Compliance compute API: 99.9% uptime target
  • Graceful degradation: if compute API is down, preview fails; Payouts creation of non-TDS payouts is unaffected
  • Kafka consumer: automatic retry with exponential backoff; DLQ after max retries
  • Payout creation latency target: < 500ms p95 (no change from current)
  • TDS Payment creation (async): Kafka publish → entity creation < 5 seconds

9.3 Security

AspectImplementation

  • No PII in logs or Kafka messages

  • Payouts API: Existing authentication (no change)
  • TDS Payment query API: Merchant can only access their own records
Kafka Security
  • Use existing Kafka authentication and authorization
  • Topic ACLs to restrict publisher/consumer access

9.4 Compliance

  • TDS Payment entities retained indefinitely for tax filing and audit
  • Payout details (TDS fields) retained per existing Payouts retention policy (7+ years)
  • Kafka messages: minimum 7-day retention
  • PAN data: encrypted at rest in Tax-Compliance; never stored in Payouts or Kafka messages

9.5 Reliability

  • Outbox pattern for Kafka publish — at-least-once delivery guaranteed
  • Daily reconciliation job: detect BB+ payouts with no corresponding tds_payments entity and alert ops
  • Payouts → Tax-Compliance are fully decoupled — failure in Tax-Compliance does not affect payout creation

9.6 Infra Cost

  • New Kafka topic: minimal cost (messages are lightweight)
  • Additional Tax-Compliance Kafka consumer: standard pod sizing
  • No new database tables in Payouts — storage via existing payout_details JSON column
