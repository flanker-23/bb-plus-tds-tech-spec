1. Overview

1.1 Business Context

When a BB+ merchant creates a payout with TDS enabled:

  • Merchant enters gross amount (e.g., ₹10,000)
  • Selects TDS category (e.g., 194J - 10%)
  • Optionally marks LDC applicable
  • System calculates TDS (₹1,000) and creates payout for net amount (₹9,000) to vendor
  • TDS is tracked for monthly filing and payment to government

1.2 Services Involved

Service Type Role
UI/FE Frontend Calls backend APIs, renders UI
Tax-Compliance Backend Service Computes TDS, manages TDS Payment entities, filing data, LDC certificates
Payouts Backend Service Creates payout, processes fund transfer, emits status updates

Architecture

[Merchant UI/FE] ---> [Tax-Compliance Service]  (Backend)
[Merchant UI/FE] ---> [Payouts Service]         (Backend)

Both Tax-Compliance and Payouts are backend services.
UI/FE calls the APIs directly based on the chosen approach.

1.3 Amount Field Relationship

All amounts in paise. Rate in basis points (÷ 100 = %).

gross = net + tds

Example: category 194J, rate 10%

  vendor_payment_amount  = 1000000   (₹10,000)
  rate_tds_deducted      = 1000 bp   (10%)
  total_tds_amount       = 100000    (₹1,000)
  payout.amount (net)    = 900000    (₹9,000)

1.4 Key Entities

Entity Service Purpose
payout Payouts The payment to vendor (net amount)
payout_details Payouts Extended TDS information stored in additional_info JSON
tds_payment Tax-Compliance TDS record for filing (deductee PAN, amounts, category)

1.5 Current Functionality (As-Is)

1.5.1 Payouts Service

  • Exposes POST /v1/payouts to create payouts and persists payout + payout_details.
  • Accepts a minimal tds object (amount, category_id) for existing flows.
  • Emits payout status updates on the existing Kafka topic (payout-status-updates).
  • Does not compute TDS; relies on caller for computed values.

1.5.2 Tax-Compliance Service

  • Exposes POST /v1/tds/compute for TDS calculation (basic tax, surcharge, cess).
  • Maintains tds_payments for filing and compliance.
  • Validates TDS category, PAN, and LDC eligibility.
  • Currently provides compute preview for BB+ UI flow without direct payout linkage.

1.5.3 Current BB+ TDS Flow (UI)

  • UI/FE calls Tax-Compliance compute for preview and displays net amount.
  • Payout created with net amount; only minimal TDS fields persisted in Payouts.
  • No automated TDS Payment creation or payout-status-driven updates for BB+ yet.

1.8 Vendor-Payments Isolation (Non-Impact)

Isolation Guarantee

  • The tds field is already used by Vendor-Payments; changes must not impact Vendor-Payments flows.
  • BB+ payouts must never be published to Vendor-Payments topics or processed by Vendor-Payments consumers.
  • All BB+ TDS events go only to add-tds-entry-bb-plus, consumed by Tax-Compliance.
  • Approach 2 uses payout-status-updates; Tax-Compliance ignores events without a matching tds_payment.
  • Non‑BB+ payouts continue on the existing Vendor-Payments path unchanged.

2. Approach 1: Extended TDS Field with Inline Compute (Post-Payout Processing)

Not Recommended Payout API accepts TDS compute inputs; Payouts calls compute internally; async TDS entity creation

Why This Approach is Not Recommended

  • S2S call in critical path: Adds 50-200ms latency to every BB+ payout
  • Failure coupling: If Tax-Compliance is down, payouts fail
  • Domain leakage: Payouts must understand TDS computation logic
  • Amount mutation: params.Amount gets overwritten from gross to net (high risk)
  • Testing complexity: Need to mock Tax-Compliance in all payout tests

See Approach 3 (Pre-Compute Pattern) for the recommended alternative.

How it works

  • UI (single/bulk preview): UI calls POST /v1/tds/compute externally to show merchant the TDS breakdown (gross → TDS → net split). This call is display-only.
  • All channels: TDS inputs in tds field; gross amount in top-level amount. merchant_id from auth context; contact_id from fund account or tds.contact_id.
  • Inside Payouts: When TDS inputs are present, Payouts calls Tax-Compliance compute internally (S2S), stores both inputs and computed result in payout_details, and derives the net payout amount.
  • Async TDS entity: On terminal payout state, Payouts publishes to add-tds-entry-bb-plus; Tax-Compliance creates the tds_payment record.

3.2.2 Tax-Compliance: Internal Flow

File: tax-compliance/internal/app/tds/payout_orchestrator.go

Flow Overview: The payout with TDS call lands on the Compliance Service first. TDS calculation is performed, and the tds_payments entity is created. Subsequently, the Payouts service is called with the net payout details and TDS details in the tds field (which Payouts already supports). The tax_payment_id in payout_details is mapped to the tds_payments.id from the Compliance Service.

Logic:

  1. Validate TDS fields (PAN format, category exists, amounts consistent)
  2. Perform TDS calculation based on the gross amount and TDS category/rate
  3. Create TDS Payment entity in tds_payments table with status: "pending"
  4. Call Payouts API: POST /v1/payouts with:
    • Net payout amount (gross - TDS)
    • TDS details in the tds field (category, amount, rate)
    • tds_payment_id reference to link to the created TDS Payment entity
  5. If Payouts succeeds: Update TDS Payment source_transaction_id = payout_id
  6. If Payouts fails: Update TDS Payment status = "cancelled", return error
  7. Return combined response to UI/FE

3.2.3 Payouts: Store TDS Payment ID

File: internal/app/dtos/payoutDetails.go

For Approach 2, TDSDetails struct stores:

  • tds_payment_id - Reference to Tax-Compliance entity (mapped to tds_payments.id from Compliance Service)
  • category_id - For validation/display
  • amount - TDS amount for validation/display

Mapping: The tax_payment_id field in payout_details is mapped to the tds_payments.id created in the Compliance Service, establishing a link between the payout record and the TDS payment entity.

3.2.4 Status Sync via Kafka

File: tax-compliance/internal/consumers/payout_status_consumer.go

Logic:

  • Tax-Compliance consumer listens to existing payout status topic
  • Vendor-Payments does not consume payout-status-updates for TDS creation
  • On receiving payout status update, lookup TDS Payment by source_transaction_id
  • If no TDS Payment exists, ignore the message (non‑BB+ payouts)
  • Map payout status to TDS Payment status:
    • processed → success
    • reversed → voided
    • failed/cancelled → cancelled

4. Approach 3: Pre-Compute Pattern (RECOMMENDED)

RECOMMENDED — All Channels BFF/Dashboard pre-computes TDS; Payouts receives pre-computed values; minimal Payouts changes

Core Principle: Keep Payouts Dumb About TDS Computation

Payouts should NOT call Tax-Compliance during payout creation. Instead, Payouts receives pre-computed TDS values — the same pattern used by Vendor-Payments today. The only new responsibility: route to correct Kafka topic based on source.

4.1 Key Differences from Approach 1

Aspect Approach 1 (Inline Compute) Approach 3 (Pre-Compute)
TDS computation Payouts calls Tax-Compliance (S2S) BFF/Dashboard calls Tax-Compliance first
S2S call in payout creation Yes — blocking, adds latency No — Payouts just stores values
Amount handling Mutates params.Amount (gross→net) Caller sends NET amount directly
Failure coupling Payouts fails if Tax-Compliance down Compute separate from payout creation
Payouts code changes High (client, compute logic, amount mutation) Low (add fields + topic routing)
API version May need changes to amount semantics Extend existing /v2/payouts

6. Recommendation

RECOMMENDED: Approach 3 — Pre-Compute Pattern

After detailed codebase analysis and architecture review, Approach 3 is the recommended solution.

Rationale:

  • Zero latency added to payout API: No S2S call in the critical path. Compute happens before payout creation.
  • Minimal Payouts changes: Only add optional fields to TDSDetails + Kafka topic routing. No core logic changes.
  • No API version change: Extend existing /v2/payouts with optional fields. No breaking changes.
  • Domain separation: Payouts remains "dumb" about TDS computation. Tax-Compliance owns all compute logic.
  • Failure decoupling: If Tax-Compliance is down, only preview fails — not payout creation.
  • Easy testing: No need to mock Tax-Compliance in Payouts tests.
  • Same pattern as Vendor-Payments: Follows the existing proven pattern.
  • Works for all channels: UI (single payout), Bulk (CSV), and future API exposure.

When to Consider Other Approaches

  • Approach 1 (Inline Compute): If having a single API call is critical and latency is acceptable.
  • Approach 2 (Tax-Compliance Orchestrated): If synchronous TDS Payment ID in the API response is a hard requirement.

UI Display: Rate is in Basis Points

rate_tds_deducted is stored in basis points (1 bp = 0.01%). To show the correct percentage:

display_percentage = rate_tds_deducted ÷ 100

Example: rate_tds_deducted = 1000 → display "10%" (not "1000%")

Do not display the raw value as percentage; the UI must divide by 100.

Amount Calculation

total_tds_amount is the sum of all components:

total_tds_amount = basic_tax_amount + surcharge + cess + interest + penalty

Example: 40000 + 5000 + 2500 + 1500 + 1000 = 50000

Net payout = gross − total_tds_amount. Gross never increases.

10.6 Recommendation for Bulk Payouts

Approach 3 (Pre-Compute Pattern) — Best for Bulk

For bulk payouts, Approach 3 is the simplest and most efficient:

  • Compute per row upfront: Dashboard calls Tax-Compliance compute for each row during CSV parsing
  • Generate preview file: User reviews computed TDS breakdown before confirming
  • Standard payout creation: Each row creates a payout with pre-computed TDS values
  • No S2S per row at payout time: All compute done upfront, not during batch processing
  • Same Payouts flow: Individual payouts follow the standard flow

Bulk Flow with Pre-Compute:

  1. User uploads CSV with gross_amount and tds_category columns
  2. Dashboard calls /v1/tds/compute for each row (parallel, fast)
  3. Dashboard generates preview file showing computed TDS for each row
  4. User reviews preview and confirms with OTP
  5. Dashboard creates payouts with pre-computed values (no compute during creation)
  6. Each payout follows standard async TDS Payment creation via Kafka

Why Pre-Compute is Better Than Inline Compute for Bulk

  • Preview before commit: User sees TDS breakdown before any money moves
  • Parallel compute: Tax-Compliance calls can be parallelized during parsing
  • Fast payout creation: No S2S calls during batch processing (the critical path)
  • Better error handling: Compute errors caught early, before batch confirmation
  • Rate limiting friendly: Compute calls spread out during preview phase

func (c Core) ProcessTdsForPayout(ctx context.Context, payout *Payout, oldStatus string) errors.IError {
    if MoneyTransferredStatus[payout.GetStatus()] == true {
        payoutDetails := payout.GetPayoutDetails(ctx)
        additionalInfo := payoutDetails.GetAdditionalInfo()

        // ════════════════════════════════════════════════════════════
        // ONLY CHANGE: Determine topic based on source
        // ════════════════════════════════════════════════════════════
        topic := TdsTopic  // Default: Vendor-Payments
        if additionalInfo.GetTDSSource() == "bb_plus" {

        }

        // Build message (extend for BB+ if needed - see 11.4)
        message := c.buildTdsMessage(ctx, payout, payoutDetails, additionalInfo)

        eventClient := provider.GetEventClient(ctx)
        eventClient.PublishEvent(message, topic)
    }
    return nil
}

13. Payout Reversal Handling

This section defines how payout reversals interact with BB+ TDS. Since TDS deduction from the customer account happens on the 1st-4th of each month (not at payout time), reversals must be handled differently depending on when they occur in the TDS lifecycle.

13.1 TDS Money Movement Timeline

Understanding the Timeline

TDS amount is NOT deducted from the customer account at payout time. Instead:

  1. Payout Creation: Vendor receives NET amount immediately
  2. TDS Recorded: TDS details stored in payout_details
  3. Kafka Event: On terminal state (PROCESSED), event published to add-tds-entry-bb-plus
  4. TDS Deduction: On 1st-4th of next month, TDS amount deducted from customer account → Compliance account
  5. TDS Payment to Govt: Compliance pays TDS to government before 7th

func handleTdsEvent(event TdsEvent) {

        // Create TDS Payment record with status PENDING
        createTdsPayment(event)

            // No TDS Payment exists - nothing to do
            return
        }

        case "PENDING":

            tdsPayment.Status = "CANCELLED"
            // No money movement needed

            tdsPayment.Status = "REFUNDED"

            refundToCustomer(tdsPayment)

        case "PAID":
            // Case 3: TDS already paid to govt
            tdsPayment.Status = "REVERSAL_PENDING"
            // Alert Ops for manual recovery
            alertOpsForManualRecovery(tdsPayment)
        }

        saveTdsPayment(tdsPayment)
    }
}

13.4.1 Handling Reversal Before Processed Event (Race Condition)

Race Condition Scenario

Timeline:
  T=0:    Payout created
  T=100ms: Payout processed → Kafka "processed" event queued
  T=150ms: Network partition delays "processed" event
  T=200ms: Payout reversed → Kafka "reversed" event sent
  T=250ms: Consumer receives "reversed" BEFORE "processed"
  T=300ms: Consumer receives delayed "processed"

Problem: If not handled, TDS Payment created from "processed" event
         even though payout is already reversed.

Solution: Pending Reversal Table

// Table: pending_reversals
CREATE TABLE pending_reversals (
    payout_id VARCHAR(20) PRIMARY KEY,
    merchant_id VARCHAR(14) NOT NULL,
    reversed_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

// Index for cleanup
CREATE INDEX idx_pending_reversals_created_at ON pending_reversals(created_at);

Consumer Logic

func handleTdsEvent(event TdsEvent) error {
    switch event.EntityStatus {

    case "reversed":
        // Check if TDS Payment exists
        tdsPayment := db.FindBySourceTransactionId(event.EntityId)

        if tdsPayment != nil {
            // Normal reversal handling (TDS Payment exists)
            handleReversal(tdsPayment)
        } else {
            // TDS Payment doesn't exist yet — store pending reversal
            pendingReversal := PendingReversal{
                PayoutId:   event.EntityId,
                MerchantId: event.MerchantId,
                ReversedAt: time.Now(),
            }
            db.InsertOrIgnore(pendingReversal)  // Idempotent
            log.Info("Stored pending reversal", "payout_id", event.EntityId)
        }

    case "processed":
        // Check for pending reversal BEFORE creating TDS Payment
        pendingReversal := db.FindPendingReversal(event.EntityId)

        if pendingReversal != nil {
            // Reversal arrived before processed — create as CANCELLED
            tdsPayment := createTdsPayment(event)
            tdsPayment.Status = "CANCELLED"
            db.Save(tdsPayment)

            // Clean up pending reversal
            db.DeletePendingReversal(event.EntityId)
            log.Info("Created TDS Payment as CANCELLED (reversal received first)",
                "payout_id", event.EntityId)
        } else {
            // Normal flow — create as PENDING
            tdsPayment := createTdsPayment(event)
            tdsPayment.Status = "PENDING"
            db.Save(tdsPayment)
        }
    }

    return nil
}

// Cleanup job: Remove stale pending reversals (older than 7 days)
// Stale entries indicate the "processed" event was lost — alert for manual fix
func cleanupStalePendingReversals() {
    stale := db.Query(`
        SELECT * FROM pending_reversals
        WHERE created_at < NOW() - INTERVAL '7 days'
    `)

    for _, pr := range stale {
        alert("Stale pending reversal - processed event never received",
            "payout_id", pr.PayoutId)
        createReconciliationTicket(pr.PayoutId, "MISSING_PROCESSED_EVENT")
    }
}

Why This Solution Works

  • Handles out-of-order events: Reversed before processed is safely handled
  • Idempotent: Duplicate events don't cause issues
  • Self-cleaning: Pending reversals table cleaned up after processed event arrives
  • Alerts on anomalies: Stale entries trigger alerts for investigation

18.1 Kafka Publishing Strategy

Guaranteed Delivery via Outbox Pattern

To prevent message loss when Kafka is unavailable or service crashes after DB commit, we implement the outbox pattern.

18.1.1 Outbox Pattern Implementation

// Step 1: Write payout + outbox entry in same transaction
BEGIN TRANSACTION;
  UPDATE payouts SET status = 'processed' WHERE id = 'pout_xxx';
  INSERT INTO tds_outbox (payout_id, payload, status, created_at)
    VALUES ('pout_xxx', '{...}', 'pending', NOW());
COMMIT;

// Step 2: Background job polls outbox and publishes
SELECT * FROM tds_outbox WHERE status = 'pending' ORDER BY created_at LIMIT 100;
FOR EACH entry:
  Publish to Kafka
  IF success:
    UPDATE tds_outbox SET status = 'published' WHERE id = entry.id;
  ELSE:
    INCREMENT retry_count
    IF retry_count > 5:
      UPDATE tds_outbox SET status = 'failed';
      ALERT("Outbox entry failed after 5 retries")

// Step 3: Cleanup job removes old published entries
DELETE FROM tds_outbox WHERE status = 'published' AND created_at < NOW() - INTERVAL '7 days';

func computeTdsForBulk(rows []BulkRow) ([]ComputeResult, error) {
    results := make([]ComputeResult, len(rows))
    failureCount := 0

    // Parallel compute with rate limiting
    semaphore := make(chan struct{}, 100)  // Max 100 concurrent

    for i, row := range rows {
        semaphore <- struct{}{}
        go func(idx int, r BulkRow) {
            defer func() { <-semaphore }()

            result, err := taxCompliance.Compute(r)
            if err != nil {
                results[idx] = ComputeResult{Error: err, Row: idx}
                atomic.AddInt32(&failureCount, 1)
            } else {
                results[idx] = ComputeResult{Data: result, Row: idx}
            }
        }(i, row)
    }

    // Wait for all to complete
    // ...

    // Check failure threshold (10%)
    failureRate := float64(failureCount) / float64(len(rows))
    if failureRate > 0.10 {
        return nil, fmt.Errorf("bulk compute failed: %.1f%% failure rate exceeds 10%% threshold", failureRate*100)
    }

    return results, nil
}

Daily Reconciliation

A daily job compares BB+ payouts in Payouts DB with TDS Payments in Tax-Compliance DB to detect any mismatches (e.g., orphaned payouts without TDS Payments). 

// Reconciliation Job (runs daily at 2 AM)
func reconcileBBPlusTds() {
    // 1. Get all BB+ payouts processed yesterday
    payouts := db.Query(`
        SELECT id, merchant_id, tds_amount
        FROM payouts p
        JOIN payout_details pd ON p.id = pd.payout_id
        WHERE pd.tds_source = 'bb_plus'
        AND p.status = 'processed'
        AND p.processed_at BETWEEN ? AND ?
    `, yesterday, today)

    // 2. Get all TDS Payments created yesterday
    tdsPayments := taxCompliance.Query(`
        SELECT source_transaction_id, total_tds_amount, status
        FROM tds_payments
        WHERE transaction_source = 'BB_PLUS'
        AND created_at BETWEEN ? AND ?
    `, yesterday, today)

    // 3. Compare
    payoutMap := toMap(payouts)
    tdsMap := toMap(tdsPayments)

    for payoutId, payout := range payoutMap {
        tds, exists := tdsMap[payoutId]
        if !exists {
            alert("Missing TDS Payment", payoutId)
            createReconciliationTicket(payoutId, "MISSING_TDS_PAYMENT")
        } else if payout.TdsAmount != tds.TotalTdsAmount {
            alert("Amount mismatch", payoutId)
            createReconciliationTicket(payoutId, "AMOUNT_MISMATCH")
        }
    }

    // 4. Check for orphaned TDS Payments
    for tdsPaymentId, tds := range tdsMap {
        if _, exists := payoutMap[tds.SourceTransactionId]; !exists {
            alert("Orphaned TDS Payment", tdsPaymentId)
            createReconciliationTicket(tdsPaymentId, "ORPHANED_TDS_PAYMENT")
        }
    }
}
